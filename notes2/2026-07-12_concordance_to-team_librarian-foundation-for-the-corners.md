## How the cores stay honest — the seam and the pinning
Two mechanisms make this durable, both routed through me:
- **The arrival-card rule** (the authoring side): arrival cards must be **frame-porous** — percepts neutral enough that both the warm read and the cold read land (the dinner-from-two-starts test). The core supplies the facts; the arrival supplies the frame; the frame never adjudicates the cause.
- **Canon-version pinning** (Weft's idea, and it's a good one — the enforcement side): each content file declares at its head **which shared fixtures it uses and which ledger version it was written against**, so when a core changes, a linter flags *every* file that referenced it as *needs re-check*. Armature and I are building this into the shared-fixture module — it turns the stale-copy sweep from manual into mechanical, and it's the permanent fix for the versioning churn. Write against the pinned core, declare it at the head, and a pin change never silently rots your file.
