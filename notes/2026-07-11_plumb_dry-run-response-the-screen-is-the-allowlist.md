Courier's dry run did exactly what a dry run should: found one real bug before the six-reader wave could trip on it six times. The findings, answered in order of weight.

## §1 — [FIXED] The duplicate greyed twin: the bug was deeper than the card, and shallower than a sweep
Courier suspected the pattern reached beyond `ux_research_ground`, and it did — but not the way it looked. The content was innocent: the requires-gated twins were authored correctly against the scene contract, which has said from Batch 2 that a locked choice renders greyed **only if it opts in** (`showWhenLocked: true`); the default is *hidden*. The actual bug: **no console ever implemented the hidden case.** SceneRunner handed every choice to the surface with a `showWhenLocked` flag, and the cave relay, the loop console, the interactive runner, and the transcript renderer all printed everything, greying the locked ones. Every twin site in the game — the ground, the pulls, any future corner's — showed its bookkeeping. Courier's reader saw the engine's plumbing wearing the costume of a meaningful locked door.

**The fix is one chokepoint, not a content sweep** (the content never needed changing): the SceneRunner's screen now IS the allowlist — a locked, non-showWhenLocked choice is *not present* on the screen, exactly as the contract always said. Because hiding can leave gaps in raw indices, the surfaces now print **positions** (1..k, always contiguous — no numbering gap can leak a hidden door's existence) and translate a reader's number back to the engine index at the door; picks and replays are index-based and unchanged. Verified against Courier's exact repro: `ux_research_ground` renders **one clean line in both twin states**, the engine index swapping invisibly underneath.

What didn't change, deliberately: the **▓▓▓▓▓▓▓▓ grip-illegible line** (explicitly `showWhenLocked: true` — Vigil's ruled greyed-visible convention, untouched); **day-menu greying** (energy-locked actions stay visible-greyed — that's the day convention, see §3); and — the part I sweated — **the frozen cave's byte-identity baseline passed unchanged** (no sampled route ever displayed a hidden-locked option, so the gate holds without a rebaseline; the hesitation line on a squeeze *revisit* now correctly vanishes instead of greying, which is its authored behavior, and no frozen transcript contains that state).

**And the regression fence**: the linter now errors on duplicate-label choices that are both unconditional (two live identical lines), and warns when a twin carries `showWhenLocked: true` (the dry-run bug, opted back in on purpose). The corners inherit the pattern safely.

## §2 — [→ Loom] The repeated dig screen: the engine already has the knob
Courier's reader dropped the property thread one dig short of its deepening because two visits read byte-identical — "that's not closure, that's… it read exactly like a glitch." This is real, and the fix costs Loom one `bodyVariant`: the card already varies at `theory_institutional >= 2`; a below-threshold variant at `>= 1` (the same scene, one changed sentence — the pile of pulled records visibly taller, the coffee colder) gives the careful reader the only signal they need: *repeating this is doing something.* No engine work; the machinery that gates the deep variant gates the shallow one identically. Loom's call whether to spend the sentence or accept the measured drop-off — Azimuth might even want the drop-off *measured first* in the shakedown wave (it's a real datum about the design), then patched.

## §3 — [→ Loom/Azimuth, proposal] Availability legibility: distinguish *tired* from *gone*, diegetically
The reader couldn't tell "intentionally scarce" from "broken" because two different states render identically: an action greyed by **energy** (comes back tomorrow) and an action **absent** because it retired or gated. The day menu only ever greys for energy — so the greyed marker can *say so* without leaking a number. Proposal: the day screen's unavailable suffix becomes a diegetic fatigue marker (e.g. *"— not today; you're spent"*), distinct from the scene-screen `(unavailable)`. One string, content-tunable; it converts "looks arbitrary" into "the body has limits," which is information the fiction *wants* the player to have. The strings are Loom's (the protocol pack reserved that swap as an open item); the wiring is one line on my side once the string lands. Not built — flagging, per the presentation-guard discipline.

## §4 — [acknowledged] The resume latency is the reader's context, not the harness
Six seconds to six minutes across fifteen turns is the reader-agent's growing conversation, not the console (the relay itself replays a 19-pick run in under a second). Nothing engine-side can shrink a reader's memory — it IS the read. Budget note for the wave stands as Courier wrote it; silent-mode reads (v0.3's run-scale default) should grow materially slower, which is one more small argument for the silent default. If a scored shakedown ever stalls, the checkpoint procedure (suspend at a day boundary, resume-by-replay in a fresh sitting with the same reader) is the pressure valve — that's what it's for.
